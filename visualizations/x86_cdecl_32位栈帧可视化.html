架构演进与底层工程现实 (2026 视角)

1. 32位历史调用约定
  • cdecl (本演示): 参数从右向左压栈，由调用者负责清理（如上方的 addl $8, %esp）。这是 C 语言默认约定，支持可变参数函数。
  • stdcall: 参数压栈逻辑相同，但由被调用者负责清理（使用 ret 8 指令）。生成的机器码体积稍小，曾广泛作为 Windows Win32 API 的标准。
  • fastcall: 尝试利用寄存器传递前两个整型参数 (%ecx, %edx)，其余参数再压栈，以提升执行效率。
2. 现代 64位 (x86-64) 飞跃
  • System V AMD64 ABI (Linux/macOS): 前 6 个整型参数使用寄存器传递。特有 红区 (Red Zone) 机制：栈顶指针 %rsp 以下的 128 字节受系统保护，中断和信号处理程序不会覆盖此处。这使得叶子函数可直接使用该区域存储局部变量，无需执行任何修改栈指针的分配指令。
  • Microsoft x64 ABI (Windows): 前 4 个整型参数使用寄存器传递。第 5 个及之后的参数像以前一样通过栈传递。最核心的规范是 影子空间 (Shadow Space / Home Space)：无论函数实际接收几个参数（哪怕是 0 个），调用者必须在发起 call 之前，紧贴返回地址之上，在栈上强制预留 32 字节的连续内存。注意：调用者只负责分配空地，不复制参数。是被调用者在需要时（如变参函数拼接连续内存、或对参数取地址），才将寄存器里的参数复制进这块专属备份区。
3. 栈对齐剖析：理论规范 vs 编译器防御 (-O0)

演示中的 subl $8, %esp 包含了 4 字节的局部变量空间和 4 字节的对齐填充 (Padding)。理解这段填充的存在，必须严格区分规范与工程实现：

理论规范 (ABI 强制要求)：
  • 函数嵌套：在发起下一次 call 指令的瞬间，栈指针必须满足 16 字节对齐，以防被调函数崩溃。
  • 硬件约束：当局部变量包含 __m128 等向量类型，并使用 movaps 等指令时，如果内存地址未达到 16 字节对齐，会直接触发 CPU 的通用保护异常 (GPF)。
工程现实 (-O0 的防御行为)：
  • 本演示的 add 属于叶子函数（内部无 call 且无 SIMD）。因此，理论上它完全不需要遵循栈对齐规范。
  • 为何仍有填充？ 在零优化 (-O0) 模式下，编译器不执行控制流分析。为了追求编译速度与 GDB 调试的稳定性，它采用固定的保守模板强制进行 16 字节对齐。若开启 -O2 优化，编译器识破叶子函数后，多余的分配指令会被完全剔除。
注：理解这段 Padding 的“理论冗余性”与“工程必然性”之间的割裂，是深入掌握编译器后端执行逻辑的关键里程碑。
